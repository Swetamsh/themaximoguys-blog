MAS 9 is 3 paradigm shifts in one platform move.

Not 3 new tools. 3 new ways of thinking.

---

**Shift 1: From "Modify" to "Extend"**

OLD WAY: You reached into IBM's code and changed it directly — custom MBOs, overridden classes, JSP modifications. You owned the full implementation. Every upgrade was a reconciliation exercise between your changes and IBM's.

NEW WAY: You build on top of the platform using supported extensibility points. Your code has clean boundaries. IBM's code has clean boundaries. Upgrades don't blow up your customizations because you didn't touch what you shouldn't have.

Feels constraining at first. Feels liberating after your first smooth upgrade.

---

**Shift 2: From "Scheduled" to "Reactive"**

OLD WAY: CRON tasks ran nightly. Integrations polled for changes. Reports executed on a schedule. The system processed in batches and the world waited for the next window.

NEW WAY: Systems react to events in real-time. A work order is created — downstream systems know immediately. A sensor reading crosses a threshold — an alert fires now, not at the next batch cycle. The question changes from "when does this run?" to "what should trigger this?"

Batch thinking is deeply ingrained. Unlearning it takes deliberate effort.

---

**Shift 3: From "Control" to "Collaborate"**

OLD WAY: You owned every layer of the stack. WebSphere configuration, database tuning, SSL certificates, patching cycles — all yours. Total control felt like security.

NEW WAY: IBM handles platform patches. Red Hat manages Kubernetes. You operate Maximo within a shared platform and focus your energy on making the application deliver value — which is where your expertise actually matters most.

You're not losing control. You're trading low-leverage control for high-leverage impact.

---

Three shifts. Same Maximo domain knowledge underneath. Fundamentally different way of applying it.
