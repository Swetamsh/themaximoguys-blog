## Thread: What Happens to Your Custom MBOs in MAS 9?

**T1 — Hook**

Your custom MBOs won't all survive the jump to MAS 9.

Here is the honest breakdown nobody is giving you:

(thread)

---

**T2 — Category 1: Green**

Simple extensions usually migrate fine.

Field defaults. Basic validation on save. Straightforward attribute overrides.

If your MBO is doing simple things in a standard way, your migration risk is low.

Test it. Verify it. Move on.

---

**T3 — Category 2: Yellow**

Complex business logic with IBM internal API dependencies — evaluate carefully.

These MBOs work today because they know Maximo's internals. When those internals shift in MAS 9, your code feels it.

The question: can this logic move to an automation script or an external microservice?

---

**T4 — Category 3: Red**

Deep framework hooks. WebSphere-specific behavior. Heavy coupling to IBM's class hierarchy.

These need a redesign conversation — not a migration conversation.

MAS 9 changes the deployment model. WebSphere is gone. That code assumes a runtime that no longer exists.

---

**T5 — The Decision Framework**

For every MBO in the Red or Yellow bucket, ask three questions:

1. Can this logic move to an automation script?
2. Can it move to an external microservice?
3. Does MAS 9 now handle this natively — making the customization unnecessary?

Option 3 happens more often than you think.

---

**T6 — The Real Question**

Before you decide how to migrate each MBO, ask the harder question:

"Does this customization still add business value — or were we working around limitations that MAS 9 has already eliminated?"

Some of your custom code exists because 7.6 forced you to write it.

Audit first. Migrate second.

---

**T7 — Pro Tip**

Before migration day arrives:

Inventory every customization you own.
Categorize it: Green / Yellow / Red.
Document the business purpose — not just the technical behavior.

Green: plan migration.
Yellow: plan evaluation.
Red: plan redesign conversation with stakeholders now, not during cutover.
